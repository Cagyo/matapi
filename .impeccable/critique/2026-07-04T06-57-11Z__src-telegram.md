## Overall Impression
A crisp, highly scannable, and well-architected Telegram bot interface that succeeds as a serious home automation tool. Commit `39c72e0` addressed several major friction points by adding wizard escape hatches (`âťŚ Cancel`), sensor selection menus for `/mute` and `/logs`, and an interactive `/camera` dashboard. The biggest remaining opportunities are: (1) eliminating "syntax dead-ends" in the `/menu` dashboard by replacing text instructions with interactive sub-menus for Config, Features, Update, and Restart; (2) clearing or disabling stale inline keyboards after action completion to prevent chat history misclicks; and (3) adding sensor selection keyboards to `/config modify` and `/config remove`.

### [P1] Stale Inline Keyboards Left Active in Chat History
- **What**: When a user clicks an inline button (in `/menu`, `/camera`, `/mute`, or wizard steps like `cfg:cancel` / `cfg:type:` / `cfg:rm:` / `import_config:apply`), the bot sends a new reply without removing or editing the inline keyboard on the previous message.
- **Why it matters**: In Telegram chat history, active buttons remain clickable indefinitely. If a user scrolls up and clicks "Confirm" on an old `/config remove` message, "Apply" on an old `/import_config`, or a sensor button on an old `/mute` message, it can trigger accidental state changes or confusing error messages (violates Nielsen #5 Error Prevention).
- **Fix**: Call `ctx.editMessageReplyMarkup({ reply_markup: undefined })` (or update the message text to show a completion/cancellation badge like `[Cancelled]` or `[Selected: Digital]`) immediately whenever an inline button is clicked or a workflow concludes.
- **Suggested command**: `$impeccable harden` (or `$impeccable polish`).

### [P2] Memory Recall Still Required for `/config modify` and `/config remove`
- **What**: When an admin sends `/config modify` or `/config remove` without arguments, the bot replies with an error: `"âťŚ Usage: /config modify <sensor_name>"`.
- **Why it matters**: While `/mute`, `/unmute`, and `/logs` were upgraded to show clickable sensor selection keyboards when called empty, `/config modify` and `/config remove` still force admins to memorize and type exact sensor names (violates Nielsen #4 Consistency & Standards and #6 Recognition vs. Recall).
- **Fix**: When `/config modify` or `/config remove` is called without a sensor name (or clicked from a Config sub-menu), reply with an inline keyboard listing all active sensors as clickable buttons (`cfg:mod:<name>` and `cfg:rm:<name>`).
- **Suggested command**: `$impeccable clarify` (or `$impeccable onboard`).

### [P3] Lack of Progress Scaffolding in Multi-Step Wizards
- **What**: In the `/config add` wizard, the user is asked a series of questions (Type -> Name -> Pin -> Active -> Pull -> Severity) without indication of how many steps remain or what they previously selected.
- **Why it matters**: If chat messages get pushed up or the admin gets distracted, they lose context on their progress and what configuration they are building (violates Nielsen #1 Visibility of System Status and Cognitive Load Working Memory rule).
- **Fix**: Prefix wizard prompts with step counters and a brief summary of current selections (e.g., `Step 3 of 6 (Digital Sensor: "front_door")\n\nGPIO pin number?`).
- **Suggested command**: `$impeccable onboard` (or `$impeccable clarify`).

## Persona Red Flags

### Alex (Power User)
- **Red Flag**: Still no batch/bulk operations. If Alex wants to silence all alerts during a noisy home maintenance session or house party, they must mute sensors one by one. A "Mute All" or "Quiet Mode (1h / 4h / 8h)" quick-action button on `/menu` or alert notifications is missing.

### Jordan (First-Timer)
- **Red Flag**: When Jordan opens `/menu` and clicks `âš™ď¸Ź Config` or `đź”§ Features`, expecting a guided menu, they hit a syntax text message (`Usage: /config add|modify|remove`), which feels like hitting a brick wall and forces them to figure out CLI command formatting.

### Casey (Distracted Mobile User)
- **Red Flag**: If Casey is scrolling through chat history on a mobile phone and accidentally taps an old inline button from a previous `/config remove` or `/import_config` message, the button is still live and tries to execute or errors out, creating confusion and misclick anxiety.
- **Red Flag**: If Casey initiates `/config modify` without remembering the exact sensor name, they get rejected with an error instead of a helpful list of clickable sensor buttons.

## Minor Observations
- In `/status`, adding relative timestamp badges (e.g., `(2m ago)`) alongside the absolute timestamp (`dd.MM.yyyy HH:mm`) would make data freshness instantly scannable without mental math.
- In `/menu`, adding a `"âťŚ Close"` button at the bottom of the keyboard would allow users to cleanly dismiss the dashboard when finished.
